## Completed

Closed after implementing and validating the MQTT/Home Assistant availability race fix.

### Implemented

#### home-ops

Merged PR: https://github.com/adampetrovic/home-ops/pull/2601

- Enabled Mosquitto persistence:
  - `persistence true`
  - `persistence_location /mosquitto/data/`
  - `autosave_interval 60`
- Added a chart-managed `ceph-block` RWO PVC mounted at `/mosquitto/data`.
- Added TCP liveness/readiness probes on port `1883`.
- Changed Home Assistant's Flux dependency from `zigbee2mqtt` to `mosquitto`.
- Added Mosquitto retained-state documentation/runbook.

#### hass-config

Merged PR: https://github.com/adampetrovic/hass-config/pull/12

- Added `packages/_config/mqtt_status.yaml` with retained HA MQTT status publishing on start/shutdown.
- Adam manually enabled retain for native MQTT birth/will messages in the Home Assistant UI.

### Validation

Ran a coordinated disruptive workload scaling test:

1. Scaled down `home-assistant`.
2. Scaled down `zigbee2mqtt` and `sigenergy2mqtt`.
3. Scaled down `mosquitto`.
4. Restarted in the historically bad order:
   - `mosquitto`
   - `zigbee2mqtt` + `sigenergy2mqtt`
   - `home-assistant` last

Results:

- HA retained status correctly changed `online` → `offline` → `online`.
- Mosquitto retained state survived broker restart.
- Publishers started before HA, and HA still reconstructed MQTT entities when started last.
- Baseline unavailable MQTT entities: 178.
- Final unavailable MQTT entities: 27.
- Final Sigenergy unavailable count: 0.
- Previously affected Shelly garage/downlights MQTT entities recovered.
- Remaining unavailable entities are reported offline by retained availability topics, so they appear to be real offline/stale entities rather than the original missed-availability race.

Final cluster state after testing:

- `mosquitto`: Ready, 1/1
- `home-assistant`: Ready, 1/1
- `zigbee2mqtt`: Ready, 1/1
- `sigenergy2mqtt`: Ready, 1/1
- Corresponding Flux Kustomizations and HelmReleases Ready.

### Conclusion

The original race condition is fixed for the tested bad restart order. Remaining unavailable MQTT entities should be handled as separate stale/offline device cleanup if desired.
