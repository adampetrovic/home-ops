# Land Talos node logs in Loki

Created: 2026-05-15 14:52 AEST

## Context

Talos currently exposes node/service logs through `talosctl logs` and kernel logs through `talosctl dmesg` / `talosctl logs kernel`, but those logs are not centrally retained in Loki.

The Talos v1.13 logging docs support network forwarding:

- Service logs: `machine.logging.destinations[]` with `tcp://` or `udp://` endpoints and `format: json_lines`.
- Service log records always include `msg`, `talos-level`, `talos-service`, and `talos-time`; `extraTags` can add static fields.
- Talos syslog is treated as a service, so extension/syslog messages are included in service log forwarding.
- Kernel logs: prefer a runtime `KmsgLogConfig` document with `url: tcp://...`; the `machine.install.extraKernelArgs` option (`talos.logging.kernel=...`) only takes effect on Talos upgrade / same-version upgrade.
- TCP sends newline-delimited JSON; UDP sends one JSON event per packet.
- The OneUptime Talos-to-Loki guide specifically calls out log-volume spikes during upgrades/restarts and recommends collector-side rate limiting plus filtering noisy sources.

Current home-ops logging path:

- `vector-agent` DaemonSet collects Kubernetes pod logs and forwards them to `vector-aggregator`.
- `vector-aggregator` sends Kubernetes logs and existing UniFi syslog logs to Loki.
- Loki is available at `http://loki-gateway.observability.svc.cluster.local` and Grafana already has a Loki datasource.

## Goal

Land Talos service, syslog-extension, and kernel logs from all five Talos nodes in Loki with useful low-cardinality labels and a safe rollout/rollback path.

Keep Talos log retention intentionally short (about two days) and protect Loki from upgrade/restart log storms with collector-side rate limits.

## Recommended design

Use the existing Vector topology, but receive Talos logs node-locally on each `vector-agent` and forward enriched events to `vector-aggregator`:

```text
Talos node logs
  -> tcp://127.0.0.1:6050/ on the same node
  -> hostNetwork vector-agent socket source
  -> Vector protocol to vector-aggregator
  -> Loki
```

Rationale:

- Keeps Talos config global and simple: all nodes can use `tcp://127.0.0.1:6050/`.
- Avoids a per-node Talos `extraTags.node=...` patch just to label hostnames.
- Lets `vector-agent` enrich events with `VECTOR_SELF_NODE_NAME`, so both service and kernel logs have a stable `node` label.
- Avoids depending on DNS/BGP/LoadBalancer reachability from Talos for the first hop.
- Reuses the existing Vector aggregator and Loki pipeline instead of introducing Promtail or another collector.

Known limitation: logs emitted before the `vector-agent` pod is running/listening will be dropped. This is acceptable for the first implementation; if early-boot retention becomes a hard requirement, revisit direct-to-aggregator or an external non-Kubernetes receiver.

Volume controls:

- Retain Talos logs for only `48h` initially, separate from the current cluster-wide Loki `14d` retention. If Loki stream retention is straightforward, add a `limits_config.retention_stream` rule for `{source="talos"}`; otherwise document the limitation and keep the Talos volume conservative until per-stream retention is implemented.
- Add Vector throttling before the Talos Loki sink so an upgrade/restart storm cannot dominate Loki ingestion. Prefer per-node throttling in `vector-agent` (for example, `throttle` after the Talos remap and before the agent's Vector sink); add an aggregator-side throttle as a second guard if validation shows it is safe.
- Start with a deliberately conservative but non-lossy-under-normal-conditions limit, e.g. `500` Talos events/sec per node with a 1-second window, then lower/raise after observing normal upgrade noise.
- Consider dropping or sampling debug-level Talos records only after observing actual noise; do not silently filter warning/error/kernel events in the initial rollout.

## Implementation plan

### Phase 0 — validate assumptions before changing live nodes

- [ ] Confirm `app-template` renders `hostNetwork: true` and `dnsPolicy: ClusterFirstWithHostNet` correctly for the `vector-agent` DaemonSet.
- [ ] Confirm no host process on Talos listens on TCP `6050`.
- [ ] Confirm Vector's `socket` source with `decoding.codec: json` correctly parses Talos newline-delimited JSON over TCP.
- [ ] Confirm talhelper can carry the Talos `KmsgLogConfig` runtime document declaratively. If not, document the supported fallback before rollout.
- [ ] Do not edit `talos/clusterconfig/` directly; only generate/inspect it with `task talos:generate`.

### Phase 1 — add Vector receive/enrich pipeline with no Talos senders yet

Update `kubernetes/apps/observability/vector/app/agent/helmrelease.yaml`:

- [ ] Enable host networking for `vector-agent`.
- [ ] Set DNS policy to `ClusterFirstWithHostNet` if required by app-template rendering.
- [ ] Keep existing Kubernetes log collection and privileged host mounts intact.

Update `kubernetes/apps/observability/vector/app/agent/resources/vector.yaml`:

- [ ] Add a local TCP socket source, bound to loopback on the host network:

  ```yaml
  sources:
    talos_source:
      type: socket
      address: 127.0.0.1:6050
      mode: tcp
      decoding:
        codec: json
  ```

- [ ] Add a remap transform to normalize fields and inject node metadata, e.g.:

  ```vrl
  .source = "talos"
  .cluster = "home-kubernetes"
  .node = get_env_var!("VECTOR_SELF_NODE_NAME")
  .message = del(.msg) ?? .message
  .level = del(."talos-level") ?? .priority ?? "info"
  .service = del(."talos-service") ?? "kernel"
  .stream = if .service == "kernel" { "kernel" } else { "service" }

  talos_time = del(."talos-time")
  if talos_time != null {
    .timestamp = parse_timestamp!(talos_time, format: "%+")
  }
  ```

- [ ] Add a `throttle` transform for Talos events before forwarding to the aggregator, initially around `500` events/sec per node, and make the limit easy to tune after observing real upgrade/restart noise:

  ```yaml
  transforms:
    talos_rate_limit:
      type: throttle
      inputs: ["talos_remap"]
      threshold: 500
      window_secs: 1
  ```

- [ ] Add a dedicated Vector sink for Talos events to the aggregator on a new internal port, e.g. `vector-aggregator.observability.svc.cluster.local:6030`, using the throttled Talos transform as input.

Update `kubernetes/apps/observability/vector/app/aggregator/helmrelease.yaml` and `resources/vector.yaml`:

- [ ] Add a dedicated Vector source on `0.0.0.0:6030` for Talos events.
- [ ] Prefer an internal-only ClusterIP service/port if app-template makes that straightforward; otherwise add a `talos` TCP port to the existing `vector-aggregator` Service and note that it is reachable on the LAN VIP.
- [ ] Optionally add a second aggregator-side `throttle` for Talos events if Vector validation confirms it gives useful protection without hiding normal per-node spikes.
- [ ] Add a Loki sink with low-cardinality labels only:

  ```yaml
  labels:
    source: "talos"
    cluster: "{{ cluster }}"
    node: "{{ node }}"
    stream: "{{ stream }}"
    service: "{{ service }}"
    level: "{{ level }}"
  ```

- [ ] Keep `remove_label_fields: true`, `remove_timestamp: true`, `out_of_order_action: accept`, and JSON encoding consistent with the existing Loki sinks.
- [ ] Do not use high-cardinality labels such as message text, sequence numbers, kernel clock, or raw source address.

Update `kubernetes/apps/observability/loki/app/helmrelease.yaml`:

- [ ] Add a short Talos-specific retention rule if supported by the deployed Loki chart/config, targeting `{source="talos"}` for about two days:

  ```yaml
  limits_config:
    retention_stream:
      - selector: '{source="talos"}'
        priority: 1
        period: 48h
  ```

- [ ] Keep the existing global retention for non-Talos logs unchanged.
- [ ] If the chart/config cannot express stream retention safely, document that blocker in this TODO and keep Talos rate limits conservative until it is solved.

Validation before Talos changes:

- [ ] `vector validate` the agent and aggregator configs, including the Talos `throttle` transform(s).
- [ ] `helm template` / `flux build kustomization vector -n observability --dry-run` succeeds.
- [ ] `helm template` / `flux build kustomization loki -n observability --dry-run` succeeds if Loki retention is changed.
- [ ] Server-side dry-run succeeds for rendered resources.
- [ ] After Flux applies, each `vector-agent` pod is Ready on hostNetwork and the existing Kubernetes log pipeline still lands in Loki.
- [ ] Verify Talos logs are subject to the intended short retention/rate-limit config without reducing retention for Kubernetes or UniFi syslog logs.

### Phase 2 — enable Talos service logs on one pilot node

Add a global Talos machine patch for service logs, but apply it to one pilot node first:

```yaml
machine:
  logging:
    destinations:
      - endpoint: "tcp://127.0.0.1:6050/"
        format: json_lines
        extraTags:
          cluster: home-kubernetes
```

Rollout steps:

- [ ] Create a new patch file such as `talos/patches/global/machine-logging.yaml` and add it to the global `patches:` list in `talos/talconfig.yaml`.
- [ ] Run `task talos:generate` and inspect the generated pilot node config only.
- [ ] Apply to a single worker first, e.g. `k8s-node-4` (`10.0.80.13`), with no reboot if Talos accepts it:

  ```bash
  talosctl apply-config -n 10.0.80.13 --file talos/clusterconfig/home-kubernetes-k8s-node-4.yaml --mode=no-reboot
  ```

- [ ] Verify in Loki:

  ```logql
  {source="talos", node="k8s-node-4", stream="service"}
  {source="talos", node="k8s-node-4", service="kubelet"}
  ```

- [ ] Verify `vector-agent` and `vector-aggregator` have no parse/backpressure errors.
- [ ] Confirm Loki ingestion rate and stream count remain well below limits.
- [ ] Confirm the Talos rate limit is not dropping normal steady-state logs; if drops occur outside an intentional stress test, raise the threshold or fix noisy parsing.

### Phase 3 — enable Talos kernel logs

Prefer `KmsgLogConfig` over `extraKernelArgs` so kernel log forwarding can be changed without a Talos upgrade:

```yaml
apiVersion: v1alpha1
kind: KmsgLogConfig
name: loki-vector
url: tcp://127.0.0.1:6050/
```

Rollout steps:

- [ ] Confirm exactly how talhelper should include the runtime config document in generated node configs.
- [ ] Apply the kernel log config to the same pilot worker first.
- [ ] Verify in Loki:

  ```logql
  {source="talos", node="k8s-node-4", stream="kernel"}
  ```

- [ ] Confirm kernel events parse cleanly despite their different fields (`clock`, `facility`, `priority`, `seq`, etc.).
- [ ] If `KmsgLogConfig` cannot be managed declaratively with talhelper, decide whether to:
  - use `machine.install.extraKernelArgs` and schedule a same-version Talos upgrade/reboot; or
  - ship service logs first and leave kernel logs as a documented follow-up.

### Phase 4 — roll to all nodes

- [ ] Apply the service log + kernel log config to remaining nodes one at a time.
- [ ] Watch Flux, Vector, Loki, and node health between nodes.
- [ ] Verify each node emits both service and kernel streams:

  ```logql
  count_over_time({source="talos", stream="service"}[5m]) by (node)
  count_over_time({source="talos", stream="kernel"}[5m]) by (node)
  ```

- [ ] Reboot or upgrade one non-critical worker during a maintenance window and confirm log forwarding resumes automatically after `vector-agent` is Ready.
- [ ] During that reboot/upgrade, watch Vector throttle/drop metrics and Loki ingestion to ensure the Talos-specific rate limit protects Loki without losing important warning/error events.

### Phase 5 — dashboards/alerts/runbook

- [ ] Add a Grafana dashboard or saved Explore links for Talos service/kernel logs.
- [ ] Add lightweight Loki alerts for high-signal Talos events only after observing normal noise, e.g. repeated service failures, kernel warnings/errors, OOM, disk/NIC issues.
- [ ] Document common queries:

  ```logql
  {source="talos", node="k8s-node-1"}
  {source="talos", stream="service", service="kubelet"}
  {source="talos", stream="kernel", level=~"warn|err|error|crit|alert|emerg"}
  ```

- [ ] Note the design in the observability/Talos runbook or AGENTS docs if this becomes the steady-state node logging pattern.

## Fallback design

If hostNetwork on `vector-agent` causes problems, fall back to direct Talos-to-aggregator TCP:

```text
Talos nodes -> tcp://10.0.88.8:6030/ -> vector-aggregator -> Loki
```

Tradeoffs:

- Simpler Kubernetes side, but Talos depends on the BGP LoadBalancer path and receiver availability.
- Requires either source-IP mapping in Vector or per-node Talos patches/`extraTags` to get a stable `node` label.
- Kernel logs have no `extraTags`, so source-IP mapping is likely required for useful kernel log labels.

## Acceptance criteria

- [ ] All five nodes emit Talos service logs to Loki.
- [ ] All five nodes emit Talos kernel logs to Loki, or kernel logging is explicitly deferred with a documented blocker.
- [ ] Loki labels are useful and low-cardinality: `source`, `cluster`, `node`, `stream`, `service`, `level`.
- [ ] Existing Kubernetes and UniFi syslog log ingestion remains healthy.
- [ ] Talos logs have an intentionally short retention target (about `48h`) without changing retention for unrelated logs, or a retention blocker is documented.
- [ ] Talos logs have an explicit Vector rate limit sized to absorb upgrade/restart spikes without overwhelming Loki.
- [ ] Vector has no sustained parse, connection, or backpressure errors.
- [ ] Loki ingestion limits are not breached.
- [ ] Log forwarding survives a `vector-agent` restart and resumes after a node reboot.
- [ ] No plaintext secrets are introduced.

## Rollback

- Remove/disable the Talos `machine.logging.destinations` and `KmsgLogConfig` first, then apply the generated config back to affected nodes with `--mode=no-reboot` where supported.
- Revert Vector hostNetwork/listener changes only after Talos senders are disabled, to avoid noisy connection failures.
- If hostNetwork breaks `vector-agent`, immediately revert the Vector change and keep Talos logging unapplied.
- If Loki ingestion spikes, lower the Talos throttle threshold or temporarily disable Talos senders before tuning filters/rate limits.
- If the Talos retention rule affects unrelated streams, revert the Loki retention change immediately and keep only the Vector-side rate limit.
